# ISB 문의 운영 및 보안안

## 목표

ISB 사이트 문의 기능은 아래 세 가지를 동시에 만족해야 합니다.

1. 고객이 쉽게 문의할 수 있어야 한다.
2. 운영자는 놓치지 않고 처리할 수 있어야 한다.
3. 스팸, 오발송, 정보유출 위험을 낮춰야 한다.

## 권장 구조

### 추천 흐름

`홈페이지 문의 폼`  
-> `Cloudflare Turnstile`  
-> `서버리스 함수(Netlify Function 또는 유사 구조)`  
-> `메일 전송 서비스(Resend)`  
-> `공용 메일함(contact@...) + 백업 수신자`

이 구조를 추천하는 이유는 다음과 같습니다.

- 정적 사이트 구조를 유지할 수 있다.
- 서버를 따로 관리하지 않아도 된다.
- 스팸 방지, 전송 기록, 주소 변경이 비교적 쉽다.
- 회사 메일과 연결하기 좋다.

## 문의 항목

### 최소 권장 필드

- 회사명 또는 이름
- 연락처
- 이메일
- 문의 종류
- 문의 내용

### 선택 필드

- 행사명
- 행사 장소
- 행사 일정
- 필요한 시스템
  - 트러스
  - G-TLD
  - HASTATI
  - SYMA
- 첨부자료 링크

### 피해야 할 것

- 주민등록번호
- 불필요한 상세 개인정보
- 과도한 파일 업로드

## 보안 설계

### 1. 입력값 검증

- 이메일 형식 검증
- 연락처 길이/문자 검증
- 너무 긴 본문 차단
- HTML/script 입력 무해화

### 2. 스팸 방지

- Cloudflare Turnstile 적용
- 동일 IP/브라우저의 반복 전송 제한
- 허니팟 필드 또는 시간 기반 방어 추가 가능

### 3. 메일 보안

- 발신 도메인에 SPF, DKIM, DMARC 설정
- 사이트용 메일 계정은 업무용 도메인 메일 사용
- 개인 메일 단독 수신 금지

### 4. 업로드 보안

파일 업로드를 나중에 붙인다면 아래 조건이 필요합니다.

- 허용 확장자 제한
- 실제 파일 타입 검사
- 파일명 재생성
- 크기 제한
- 실행권한 없는 저장 위치 사용

## 운영 규칙

### 수신 규칙

- `contact@회사도메인`
- `webmaster@회사도메인`
- 필요 시 운영자 개인 업무 메일을 참조 수신

### 응답 기준

- 영업일 기준 1일 내 1차 응답
- 견적/기술검토 문의는 내부 전달 후 상태 기록

### 기록 기준

간단한 스프레드시트 또는 노션에 아래만 남겨도 충분합니다.

- 접수일시
- 발신자
- 문의유형
- 담당자
- 답변 여부
- 후속조치

## 장애 대응

### 문의 폼이 실패할 때

- 실패 메시지를 사용자에게 명확히 보여준다.
- 대체 연락 수단을 즉시 제시한다.
  - 전화
  - 이메일
- 운영자에게 별도 장애 알림이 가면 가장 좋다.

### 메일이 누락될 때

- 주 1회 테스트 문의 발송
- 스팸함 확인
- 발신 인증(SPF/DKIM/DMARC) 재점검

## 개인정보 운영 원칙

- 문의 수집 목적을 페이지 하단에 간단히 고지
- 필요한 정보만 수집
- 오래된 문의 데이터는 보관 기준을 두고 정리
- 외부 전달 시 최소한으로 공유

## 구현 단계 제안

### 1단계

- 현재는 전화/메일 직접 노출 + 간단한 문의 폼
- Turnstile 적용
- 공용 메일 수신 연결

### 2단계

- 문의 유형별 자동 분기
- 접수 자동응답 메일
- 상태 추적 표 연동

### 3단계

- CRM 또는 영업 파이프라인 연동
- 파일 첨부 기능
- 고객 분류/후속 마케팅 연동

## 권장 기술 조합

### 가장 무난한 구성

- 정적 사이트
- Netlify Functions
- Cloudflare Turnstile
- Resend
- Microsoft 365 또는 Google Workspace

### 더 간단한 구성

- 외부 폼 서비스 사용

장점:

- 구축 빠름

단점:

- 브랜딩/제어/데이터 이전성이 떨어질 수 있음
