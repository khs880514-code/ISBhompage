# ISB 웹사이트 운영 매뉴얼

## 목적

이 문서는 ISB 회사 웹사이트를 `개인 1명이 실무 운영하더라도`, 회사 자산으로 안전하게 유지하기 위한 운영 기준입니다.

핵심 원칙은 아래 5가지입니다.

1. 공개 사이트는 단순하고 안정적으로 유지한다.
2. 계정, 도메인, 저장소는 `개인 자산`이 아니라 `회사 자산`으로 관리한다.
3. 수정 권한과 공개 사이트를 분리한다.
4. 문의는 놓치지 않도록 `공용 수신 체계`로 운영한다.
5. 문제가 생겼을 때 복구할 수 있도록 백업과 책임자를 명확히 한다.

## 권한 구조

### 필수 역할

- 운영 책임자 1명
  - 사이트 내용 수정, 문의 확인, 배포 승인
- 백업 관리자 1명
  - 책임자 부재 시 도메인/호스팅/메일 복구 가능
- 외부 협력사 또는 개발 지원자
  - 필요 시 저장소 접근 또는 디자인/배포 지원

### 계정 원칙

- 도메인, DNS, GitHub, 호스팅, 메일은 모두 회사가 회수 가능한 계정으로 만든다.
- 최소 2명은 핵심 계정 복구권한을 가진다.
- 관리자 권한은 필요한 사람에게만 준다.
- 모든 계정은 `비밀번호 관리자 + MFA`를 적용한다.
- 복구 코드와 결제 수단 정보는 별도 안전한 문서에 남긴다.

## 운영 대상 자산

### 꼭 목록화할 것

- 대표 도메인
  - 예: `e-isb.com`
- 보조 도메인
  - 예: `e-isb.co.kr`
- DNS / 보안 계정
  - 예: Cloudflare
- 소스 저장소
  - 예: GitHub 조직 또는 회사 계정
- 배포 서비스
  - 예: Netlify
- 회사 메일
  - 예: Microsoft 365 또는 Google Workspace
- 문의 발송 서비스
  - 예: Resend
- 검색/분석 도구
  - 예: 네이버 서치어드바이저, Google Search Console, GA4

## 일상 운영 흐름

### 새 프로젝트 업데이트

1. 현장 사진 정리
2. 블로그 또는 공식 소개용 글 작성
3. `admin.html` 또는 JSON 데이터로 메인 대표 사례/프로젝트 카드 수정
4. 내용 검수
5. 운영 책임자 확인 후 공개
6. 네이버/구글 수집 도구에 주요 페이지 반영 상태 확인

### 문의 처리

1. 사이트 문의 접수
2. 공용 메일함으로 수신
3. 운영 책임자 또는 영업 담당자가 1차 확인
4. 필요 시 내부 전달
5. 답변 완료 후 상태 기록

## 주간 점검

- 사이트 접속 확인
- 문의 메일 수신 테스트
- 주요 링크 깨짐 여부 확인
- 대표 전화/메일/주소 최신 여부 확인

## 월간 점검

- 도메인 만료일 확인
- 계정 MFA/권한 점검
- 백업 파일 열림 여부 확인
- 문의 응답 누락 여부 확인
- 오래된 시공 사례 정리
- 분석 도구에서 유입 검색어/방문 페이지 확인

## 분기 점검

- 복구 시나리오 점검
  - 도메인 계정 로그인 가능 여부
  - 호스팅 계정 로그인 가능 여부
  - 저장소 복원 가능 여부
- 관리자/협력사 권한 재검토
- 회사 소개/포트폴리오 방향 업데이트

## 백업 정책

### 최소 백업 항목

- Git 저장소 전체
- `site-data` JSON 내보내기 파일
- 주요 시공 이미지 원본
- 도메인/DNS 설정 스크린샷 또는 문서
- 메일 계정/별칭 구조 문서

### 권장 백업 위치

- 회사 OneDrive 또는 Google Drive
- 로컬 외장 저장소 또는 별도 PC
- GitHub 저장소

## 사고 대응

### 문의가 안 들어올 때

1. 사이트 폼 전송 테스트
2. 메일 발송 서비스 상태 확인
3. 스팸함 확인
4. 임시로 연락처 직접 노출

### 사이트가 열리지 않을 때

1. 도메인 만료 여부 확인
2. DNS 상태 확인
3. 배포 서비스 상태 확인
4. 최근 변경분 롤백

### 계정 탈취 의심 시

1. 비밀번호 즉시 변경
2. 모든 세션 종료
3. MFA 재설정
4. 토큰/API 키 재발급
5. 변경 이력 확인
6. 백업 관리자에게 즉시 공유

## 운영 정책

### 공개 페이지 원칙

- 관리자 페이지를 공개 메뉴에 노출하지 않는다.
- 공개 사이트는 정적 사이트로 유지한다.
- 서드파티 스크립트는 꼭 필요한 것만 쓴다.

### 콘텐츠 원칙

- 회사 대표 연락처는 항상 최신 상태 유지
- 대표 이미지와 설명이 서로 맞도록 유지
- 포트폴리오는 너무 오래된 것만 남기지 않도록 갱신

### 보안 원칙

- 관리자 계정 전부 MFA 적용
- 비밀번호는 길고 랜덤하게, 서비스별로 다르게 사용
- 문의폼은 스팸 방지와 입력값 검증을 반드시 적용
- 소유권 검증 가능한 검색/배포/도메인 구조 유지

## 실행 우선순위

### 1차

- 도메인/호스팅/메일/저장소 책임자 정리
- 공용 메일함 정리
- MFA 전부 적용
- 운영 문서 1차 작성

### 2차

- 문의폼 구축
- 검색도구 등록
- 배포 자동화
- 백업 자동화

### 3차

- 관리자 페이지 고도화
- 접근제어(Cloudflare Access 등) 적용
- KPI 대시보드 정리
